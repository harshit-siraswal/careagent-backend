The current MCP tool surface can list projects, query logs, run SQL, and apply
database migrations, but it does not expose built-in Auth provider mutation
tools. Supabase Auth logs confirmed the current failure as:

- Endpoint: `/authorize`
- Status: `400`
- Error code: `validation_failed`
- Message: `Unsupported provider: provider is not enabled`

This is an Auth service configuration issue, not a database/RLS issue. Google
provider secrets must be configured in the Supabase dashboard or through the
Supabase Management API with an authenticated personal access token.

## Required Dashboard Settings

In Supabase:

1. Open Authentication > Providers.
2. Enable Google.
3. Add the Google OAuth client ID.
4. Add the Google OAuth client secret.
5. Keep the provider scopes limited to OpenID, email, and profile for the MVP login flow.
6. Add `app.careagent://auth-callback` to the Auth URL allow list / additional redirect URLs.

In Google Cloud Console:

1. Configure the OAuth consent screen for CareAgent.
2. Create or update the OAuth client used by Supabase.
3. Add this authorized redirect URI:
   - `https://kgkfrrffrjfltswwcsmw.supabase.co/auth/v1/callback`

For local Supabase development, also add the local callback URI generated by the Supabase CLI if the local auth service is used.

## Management API Option

Supabase's current Google provider docs also support patching the Auth service
configuration programmatically:

```powershell
$body = @{
  external_google_enabled = $true
  external_google_client_id = $env:GOOGLE_CLIENT_ID
  external_google_secret = $env:GOOGLE_CLIENT_SECRET
} | ConvertTo-Json

Invoke-RestMethod `
  -Method Patch `
  -Uri "https://api.supabase.com/v1/projects/kgkfrrffrjfltswwcsmw/config/auth" `
  -Headers @{ Authorization = "Bearer $env:SUPABASE_ACCESS_TOKEN" } `
  -ContentType "application/json" `
  -Body $body
```

Do not commit `SUPABASE_ACCESS_TOKEN`, `GOOGLE_CLIENT_ID`, or
`GOOGLE_CLIENT_SECRET`. The Google client secret belongs only in Supabase
provider configuration.

## Flutter Runtime Configuration

The Flutter app expects public Supabase values through `--dart-define`:

```powershell
flutter run --dart-define=SUPABASE_URL=https://kgkfrrffrjfltswwcsmw.supabase.co `
  --dart-define=SUPABASE_PUBLISHABLE_KEY=<publishable-or-anon-key> `
  --dart-define=SUPABASE_REDIRECT_URL=app.careagent://auth-callback
```

Use `SUPABASE_ANON_KEY` only if the project still uses legacy anon keys. Never put a service-role key in a Flutter build.

## Backend Follow-Up

The Supabase database now includes a CareAgent auth bridge:

- `auth_provider` includes `supabase`.
- `auth.users` inserts/updates trigger `private.sync_supabase_auth_user()`.
- The trigger upserts `user_accounts` and `auth_identities`.
- RLS helper functions fall back to `auth.uid()` and JWT app metadata when the FastAPI-local `app.*` settings are absent.

The FastAPI backend still uses placeholder bearer/header auth for local contract tests. To accept Supabase-authenticated requests in production:

- Validate Supabase JWTs against the project JWKS.
- Map `auth.users.id` to `user_accounts.id` or store a Supabase Auth identity link in `auth_identities`.
- Avoid authorization decisions based on user-editable metadata.
- Keep patient access grants and role permissions in backend-owned tables.
- Add tests for token validation, revoked/inactive users, and patient-scope denial.
